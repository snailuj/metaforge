# Evolution v2 — Rotation, Survival Criteria & Expanded Fixture

**Date:** 2026-02-12
**Status:** Design complete, pending implementation

## 1. Problem Statement

Evolution v1 ran 10 trials against a fixed 49-pair evaluation fixture. Post-hoc
analysis revealed two issues:

1. **Prompt leakage** — the LLM tweak generator embedded evaluation pairs as
   prompt examples, inflating MRR (+25.4% headline collapsed to +5.5% corrected).
2. **Tacit overfitting** — even without explicit leakage, steering prompts against
   a fixed eval set risks inadvertently optimising toward the particular terrain
   of those 49 metaphors rather than generalising across the full space.

Evolution v2 addresses both problems by expanding the fixture to ~275 pairs,
rotating a subset per generation, and replacing the naive "MRR > parent" gate
with statistically defensible survival criteria.

## 2. Expanded Evaluation Fixture

### 2.1 Source

225 new pairs generated by Opus 4.6, staged in
`data-pipeline/fixtures/metaphor_pairs_generated.json`. Combined with the
existing 49 pairs (after reclassification), the full pool is ~275 pairs.

### 2.2 Domain Buckets

| Domain | Description | Target count |
|--------|-------------|:---:|
| emotion | Emotional states → physical/natural phenomena | ~40 |
| time | Time, lifecycle, mortality, seasons of life | ~30 |
| cognition | Mind, thought, knowledge, ideas, learning | ~30 |
| society | Power, justice, freedom, politics, systems | ~30 |
| body | Physical body → objects, materials, machines | ~25 |
| nature | Natural phenomena, weather, landscape as metaphor | ~25 |
| relationship | Trust, betrayal, friendship, love, social bonds | ~25 |
| creativity | Art, music, language, stories, imagination | ~20 |

### 2.3 Tier Definitions

- **Strong (~30%):** Culturally universal, well-attested. Most English speakers
  recognise instantly. Often bordering on dead metaphor. Examples: anger→fire,
  time→river, knowledge→light.
- **Medium (~40%):** Recognisable but require a moment's thought. Established in
  literature or common speech but not cliché. Examples: thought→thread,
  memory→palace, silence→blanket.
- **Weak (~30%):** Novel, culturally specific, or highly context-dependent. A
  reader might say "oh, I see it" but wouldn't generate it spontaneously. Tests
  whether prompts genuinely understand metaphorical reasoning vs pattern-matching
  canonical metaphors. Examples: bureaucracy→maze, nostalgia→amber,
  procrastination→quicksand.

### 2.4 Reclassification of Existing Pairs

The existing 49 pairs are reclassified using empirical per-pair data from
evolution v1. Tier assignment is based on observed difficulty for the system,
not cultural canonicity:

- Pairs that consistently scored 0.0 across trials (e.g. love→journey,
  freedom→bird, eye→window) are demoted from "strong" regardless of how
  canonical the metaphor is culturally.
- The per-pair RR averages from the evolution report (Section 6) provide the
  calibration data.

### 2.5 Register Tag

An optional `register` field tags pairs rooted in mythological, spiritual, or
Jungian archetypal traditions:

```json
{"source": "soul", "target": "bird", "tier": "medium", "domain": "body", "register": "archetypal"}
```

Most pairs have `"register": null`. The rotation algorithm ensures each
generation's subset includes some archetypal-register pairs.

### 2.6 Schema

```json
{
  "source": "string (abstract noun)",
  "target": "string (concrete/sensory noun)",
  "tier": "strong | medium | weak",
  "domain": "emotion | time | cognition | society | body | nature | relationship | creativity",
  "register": "archetypal | null"
}
```

### 2.7 Curation Pipeline

Before the generated pairs enter the live pool:

1. **Automated filter** flags suspicious pairs:
   - Target word not in FastText vocabulary (won't have embeddings)
   - Source word not sufficiently abstract (heuristic: appears in concrete
     noun lists)
   - Duplicate or near-duplicate targets within the pool
   - Bridge plausibility check (source and target too distant in embedding
     space to ever match)
2. **Human review** of flagged pairs — approve, edit, or reject.
3. **Domain and register tags** added to existing 49 pairs to match new schema.
4. **Merge** into single canonical fixture file.

## 3. Rotation Algorithm

### 3.1 Core Mechanism

Each generation evaluates a **subset** of the full pair pool, not the entire
pool. This prevents tacit overfitting to a fixed eval set.

- **Subset size:** ~65% of the pool (with ~275 pairs, that's ~179 per generation)
- **Tiered quotas:** The subset maintains the pool's tier distribution. If the
  pool is 30/40/30 strong/medium/weak, each subset draws proportionally:
  - ~54 strong, ~72 medium, ~54 weak (scaled to subset size)
- **Register coverage:** Each subset includes at least 2 archetypal-register pairs.

### 3.2 Controlled Overlap

To support the paired-comparison survival gate (Section 4.1), consecutive
generations share a guaranteed minimum overlap:

- **50% carry-over:** Half the pairs from generation N are guaranteed to appear
  in generation N+1.
- **15% fresh draw:** ~15% of the subset is drawn from pairs NOT in the previous
  generation's subset.
- **Remaining ~35%** fills to the target subset size, drawn randomly from the
  full pool (may or may not overlap with the previous generation).

This yields ~77% effective overlap between consecutive generations (50%
guaranteed + random overlap in the remaining draw), providing a solid base
for paired comparison while still cycling fresh pairs.

### 3.3 Carry-Over Selection

The 50% carry-over is drawn from the previous generation's subset, maintaining
tier proportions. It is NOT biased toward pairs the prompt performed well on —
carry-over is random within each tier.

### 3.4 Seeding and Reproducibility

Each generation's subset draw uses a deterministic RNG seed derived from the
experiment ID and generation number. The seed, full subset list, and overlap
with the parent are logged in the trial record (Section 6).

## 4. Survival Criteria

### 4.1 Layer 1 — Per-Generation Gate (Paired Comparison)

The immediate survive/die decision for each mutation:

1. Identify the **shared pairs** between this generation's eval subset and the
   parent generation's eval subset.
2. Compute MRR on shared pairs only for both the mutant and parent prompts.
3. **Survive** if `mutant_shared_mrr - parent_shared_mrr > epsilon` where
   `epsilon = 0.005`.
4. **Die** if delta ≤ epsilon — revert to parent prompt.

This controls for subset difficulty: two generations seeing different subsets
cannot produce misleading deltas because the comparison is restricted to pairs
they have in common.

### 4.2 Layer 2 — Cross-Lineage Ranking (Bradley-Terry / ELO)

A global ranking of all prompt variants across all lineages:

- Every trial's per-pair results feed into a Bradley-Terry model.
- Each pair in each trial is a "match" where the prompt either found the target
  (win) or didn't (loss), weighted by reciprocal rank.
- The model produces a global rating for each prompt variant.
- **Purpose:** After exploration completes, the ratings answer "which lineage
  should we invest more exploitation budget in?" This does NOT gate individual
  mutations — it's a strategic signal.
- Stabilises after ~10–15 trials across families.
- Ratings are logged after each trial for retrospective analysis.

### 4.3 What Counts as a Failure

| Condition | Classification | Counts toward consecutive limit? |
|-----------|---------------|:---:|
| Delta on shared pairs ≤ 0.005 | Regression / no improvement | Yes |
| Enrichment coverage < 90% | Infrastructure failure | No |
| Tweak generation returns empty/invalid | Tweak failure | Yes |
| All enrichment batches fail (0 properties) | Infrastructure failure | No |

Infrastructure failures are excluded from the consecutive failure counter to
prevent transient API issues from killing viable lineages.

## 5. Exploitation Stopping Conditions

### 5.1 Dynamic Consecutive Failure Limit

The tolerance for consecutive failures tightens as generations progress,
mirroring the narrowing search space:

| Generation | Consecutive failure limit | Rationale |
|:---:|:---:|---|
| 1–3 | 5 | Early mutations explore widely; combinatorics favour discovery |
| 4–6 | 3 | Search is narrowing; diminishing returns expected |
| 7–10 | 2 | If the lineage hasn't plateaued, it's close; kill fast |

### 5.2 Max Generation Cap

Hard ceiling of **10 generations** per lineage regardless of success. Reassess
after the first full experiment whether deeper exploitation is warranted.

### 5.3 Stopping Summary

A lineage stops when the **first** of these conditions is met:

1. Consecutive failure limit reached (dynamic, per table above)
2. Generation 10 reached
3. Infrastructure failure on 3+ consecutive trials (regardless of generation)

## 6. Instrumentation

Every `TrialResult` record includes:

```json
{
  "trial_id": "exploit-persona_poet-g3",
  "prompt_name": "persona_poet",
  "prompt_text": "...",
  "mrr": 0.0952,
  "mrr_shared": 0.0941,
  "parent_mrr_shared": 0.0904,
  "shared_delta": 0.0037,
  "per_pair": [...],
  "secondary": {...},
  "eval_subset": ["pair_001", "pair_002", ...],
  "shared_with_parent": ["pair_001", "pair_003", ...],
  "rotation_seed": 42031,
  "pool_version": "sha256:abc123...",
  "parent_id": "exploit-persona_poet-g2",
  "generation": 3,
  "mutation": "...",
  "survived": false,
  "timestamp": "...",
  "enrichment_coverage": 0.97,
  "valid": true,
  "elo_rating": 1523.4
}
```

New fields vs v1:

| Field | Purpose |
|-------|---------|
| `mrr_shared` | MRR computed only on pairs shared with parent |
| `parent_mrr_shared` | Parent's MRR on the same shared pairs |
| `shared_delta` | `mrr_shared - parent_mrr_shared` (the survival gate input) |
| `eval_subset` | List of pair IDs used in this generation |
| `shared_with_parent` | Pair IDs that overlapped with parent's subset |
| `rotation_seed` | RNG seed for reproducibility |
| `pool_version` | Hash of the pair pool file at experiment time |
| `elo_rating` | Bradley-Terry rating after this trial |

## 7. Implementation Sequence

1. **Curate pairs** — run automated filter on generated 225, human review
   flagged pairs, reclassify existing 49, merge into canonical fixture with
   new schema.
2. **Rotation module** — implement subset selection with tiered quotas,
   controlled overlap, seeding, and logging.
3. **Paired-comparison gate** — modify `run_exploitation()` to compute
   shared-pair MRR and apply epsilon threshold.
4. **Dynamic stopping** — implement generation-dependent consecutive failure
   limits.
5. **Bradley-Terry layer** — implement cross-lineage rating accumulation.
6. **Extend TrialResult** — add new instrumentation fields.
7. **Update report generator** — surface rotation stats, shared-pair deltas,
   and ELO ratings.
8. **Test suite** — TDD throughout; tests for rotation determinism, overlap
   guarantees, survival gate edge cases, stopping condition transitions.

## 8. Budget Estimate

With ~275 pairs, ~65% subsets, 5 exploration prompts, up to 10 exploitation
generations per survivor, worst case:

| Phase | Trials |
|-------|:---:|
| Exploration (baseline + 5) | 6 |
| Exploitation (worst case: 5 survivors × 10 gens) | 50 |
| **Total worst case** | **56** |

Dynamic stopping and consecutive failure limits typically reduce exploitation
by 40–60%, so expected total is **25–35 trials**. At ~700 synsets/trial with
Gemini enrichment, budget is comparable to v1 (~$20–25).
